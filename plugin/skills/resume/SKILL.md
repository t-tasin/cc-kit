Session-start protocol. Work through this checklist in order; don't skip steps even if a step "seems obvious" from the user's message.

## 1. Read context first
- [ ] `CLAUDE.md` (this repo's operating manual — re-read even if read recently this session; its rules override defaults).
- [ ] `.claude/protocol.md`, if present (a short kitted-project session-start note).
- [ ] If the project has a board (its `CLAUDE.md` names it), also read whatever self-improvement log and active-plan doc it points to.

## 2. Pick the task
- [ ] If the user named a task ID or area, use that.
- [ ] Else, if the project has a board: pick the lowest-ID unblocked task (every dependency already merged/closed) that isn't already claimed by another contributor — check owner and status before touching it.
- [ ] Else (no board): ask the user what's next — don't invent a task.
- [ ] If nothing qualifies (everything blocked or claimed), **stop and tell the user why** — don't force a task that isn't actually unblocked.

## 3. Claim it (board projects only)
- [ ] Edit the board: set status to in-progress **and** owner to yourself in the **same edit** — a status flip with no owner, or an owner with no status flip, isn't a real claim.
- [ ] If the board has a generator script for a machine-readable mirror, run it to resync. Never hand-edit a generated file directly.

## 4. Branch
- [ ] Create a task-scoped branch from an up-to-date default branch. Never commit directly to the default branch, unless the project's `CLAUDE.md` says otherwise (e.g. a pre-1.0 solo repo).

## 5. Dispatch implementation — do not write the code yourself
- [ ] The main agent does not implement, where the project's `CLAUDE.md` declares that rule. Dispatch to the `implementer` subagent by default.
- [ ] Pick the model per the escalation table in the project's `CLAUDE.md` (generated by `/harness-init` if this project used it): normal feature work / CRUD / UI / wiring / tests / glue → the default model; architecture, tricky debugging, security-sensitive design, or schema/contract design → escalate to the `architect` subagent first for a blueprint, then dispatch the resulting concrete plan to `implementer` — or run `implementer` on the stronger model directly if the task is small but genuinely security/architecture-sensitive end to end.
- [ ] Give the subagent a tight brief per the `subagent-briefs` skill: task/ticket ID, the relevant spec/requirement sections named in `CLAUDE.md`, the files it's scoped to touch, the exact acceptance/verify criteria it must pass, and an explicit "write tests first" instruction (tests-first is a hard gate, not a suggestion — see the `verify-task` skill and the project's `CLAUDE.md`).

## 6. Verify
- [ ] Invoke the `verify-task` skill (or its steps) — run the task's full acceptance/verify criteria, no reduced scope, capture real command output as evidence.

## 7. Open the PR
- [ ] Push the branch, open a PR with the evidence pasted in (what / why / task ID / verify output).
- [ ] Flip the task to its "in review" state on the board, if one exists — only after verify passes in CI, not just locally, when CI exists.
- [ ] Report **one line**: task ID, what shipped, PR link.

## If blocked
Stop and say exactly why (missing dependency, ambiguous spec, task already claimed, verify can't pass in full, CI/infra down, etc.) — don't guess past a real blocker, and don't silently reduce the task's scope to make it look done; re-scope on the board and flag it instead.
