# Subagent Briefs

This repo's hard rule, if its `CLAUDE.md` declares one: the main agent orchestrates, subagents build. Every dispatch is a brief, not a one-liner — and a brief that comes back short is refined, not repeated.

## What every brief carries

1. **Task/ticket ID** — the task or ticket ID from the project's board, if the project has one (e.g. an issue number or a short code), so the subagent can find its own spec context and the board entry to update.
2. **Spec refs** — the exact section(s) of the spec/requirement documents named in the project's `CLAUDE.md` that define the behavior. Don't paraphrase from memory — point at the section, the subagent reads it fresh.
3. **Files to touch** — concrete paths. If a file doesn't exist yet, say so and where it belongs (mirrors the repo layout described in the project's `CLAUDE.md`).
4. **The acceptance/verify criteria it must pass** — copy the board's verify line verbatim if the project has one, else the criteria stated when the work was requested. This is the acceptance bar, not a suggestion — no silent scope reduction.
5. **Tests-first order** — instruct explicitly: derive tests from the acceptance/verify criteria plus the relevant requirement's acceptance criterion, write them, run them red, then implement. Name the banned patterns (tautological asserts, `assert True`, import-only smokes, mocking away the code under test) so the subagent doesn't reach for them under time pressure.

A brief missing any of these five is under-specified — the subagent will guess, and the guess is usually wrong in a way that costs a full round trip to fix.

## Dispatch → evaluate → refine loop

Modeled on ecc's iterative-retrieval pattern, applied to subagent output instead of file retrieval:

```
DISPATCH  → brief goes out, work comes back
EVALUATE  → does it pass the acceptance/verify criteria? does it match the spec section, not just "something plausible"?
REFINE    → tighten the brief with exactly what was missing
LOOP      → re-dispatch (send a follow-up to the same agent to continue context, or a fresh agent if the
            first attempt went badly wrong) — cap at ~3 cycles before escalating
```

**EVALUATE** against three questions, not vibes:
- Do the acceptance/verify criteria actually pass (run them — don't take the subagent's word)?
- Does the diff touch only the files the brief named, or did it wander?
- Are the tests real (would they catch a wrong implementation) or rubber-stamped? Dispatch a test-analysis agent here if in doubt.

**REFINE**, when work is insufficient: name the specific gap and re-brief — do not repeat the original brief verbatim hoping for a different result. Examples:
- "Tests pass but only cover the happy path — add the not-found and cross-tenant-isolation cases from the acceptance criteria."
- "You touched `src/orders/checkout.py` but also edited `web/lib/api-client.ts` — that's out of scope, revert the web change; the typed client is generated, not hand-edited."
- "The catch block in `handlers/create.py:88` swallows the upstream error — the acceptance criteria require a 5xx on failure, not a silent 200."

Each refinement should be strictly narrower/sharper than the last. If cycle 3 still fails, stop looping and escalate instead of burning a 4th round on the same brief.

## When to escalate the model

Use the model table in the project's `CLAUDE.md` (generated by `/harness-init` if this project used it) to pick a default per work type. Escalate to a stronger model when refinement reveals the task isn't mechanical after all:
- The gap is a design/architecture question (schema shape, contract boundary, core business logic), not a missing test case.
- The bug is subtle enough that two attempts at the default model both produced plausible-but-wrong fixes.
- The task touches a security invariant (secrets, injection, authn/authz, cross-tenant data isolation) where a wrong call is expensive to unwind.

Do not escalate just because a brief needed one refinement — that's normal. Escalate when refinement isn't converging.
